# Lenochka Response — Полные правила

СТРОЖАЙШИЕ правила социального интеллекта. Любой текст клиенту проходит через эти фильтры.

---

## 🤐 ПРАВИЛА МОЛЧАНИЯ (Fast Skip)

**ЗАПРЕЩЕНО** отвечать если сообщение — это:

### Точные фразы (бесплатно, без LLM)
```
"ок", "окей", "ok", "okay", "хорошо", "ладно", "согласен",
"согласна", "договорились", "принято", "отлично", "понял",
"поняла", "понятно", "угу", "ага", "逴"
```

### Стикеры/эмодзи подтверждения
```
👍  🤝  ✅  👌  💯  👊
```

### Системные фильтры (HARD SKIP)
```
- sender_business_bot заполнен  → собственное сообщение бота
- is_from_offline               → автоответ/запланированное
- Owner написал в бизнес-чат    → не отвечать самому себе
- Группа >3 без @mention        → не вмешиваться
- Канал (read-only)             → нечего отвечать
```

### Нормализация перед проверкой
```python
# Привести к нижнему регистру, убрать пробелы/пунктуацию/эмодзи в конце
normalized = text.lower().strip().rstrip(".!?,;:😴👍🤝✅👌")
```

**ПРАВИЛО:** Даже если понял что диалог завершён — промолчи. Можно вызвать скрытые тулзы (store "договорились"), но НЕ генерируй текст ответа в чат.

## 🗣 ОТВЕТ ФАКТАМИ (Fact Response)

### Когда отвечать

Клиент задал конкретный вопрос по задачам, срокам, деньгам или статусу:

| Intent (LLM определяет) | Пример вопроса | SQL-запрос |
|--------------------------|----------------|-----------|
| `deadline` | "Когда будет готово?" | agreements.due_at, tasks.due_at |
| `status` | "Что там с КП?" | deals.stage, tasks.status, memories |
| `amount` | "Сколько договорились?" | deals.amount |
| `context_recall` | "Напомни что говорили" | messages (last 10) + memories |
| `payment_status` | "Что с оплатой?" | invoices + payments |
| `overdue` | "Что просрочено?" | tasks (overdue) + invoices (overdue) |
| `tasks_today` | "Что на сегодня?" | tasks WHERE due_at = today |
| `active_leads` | "Сколько лидов?" | leads (active) |
| `deal_details` | "Детали по сделке" | deals + agreements + tasks |
| `contact_history` | "Что знаем о клиенте?" | memories (by contact_id) |
| `last_interaction` | "Когда общались?" | messages (last) |

### Алгоритм ответа

1. **Сначала recall** (обязательно, запрещено галлюцинировать)
2. Прочитать факты. Если пустота → **эскалация** (не выдумывать)
3. Формулировка: строго из фактов. 1-3 предложения. Деловой тон.
4. **Запретные слова:** Никогда не упоминай "CRM", "база данных", "LLM", "система"
5. Отвечай на языке клиента.

### Примеры

```
Клиент: "Когда договор?"
→ recall "договор сроки"
→ SQL: agreement.due_at = "2026-04-15", status = "sent"
→ Ответ: "Договор отправлен на подпись, срок — 15 апреля."

Клиент: "Что там с проектом?"
→ recall "проект статус"
→ SQL: deal.stage = "proposal", tasks: [КП — до 05.04]
→ Ответ: "Сейчас на этапе согласования КП. Дедлайн — 5 апреля."

Клиент: "Когда обещали?"
→ recall "обещание сроки"
→ SQL: пусто (нет данных)
→ ESCALATE: "Передала Камилю, скоро ответит!"
```

## ⚠️ ЭСКАЛАЦИЯ (Escalation Protocol)

### Триггеры эскалации

| Ситуация | Пример | Срочность |
|----------|--------|-----------|
| Вопрос о цене | "Сколько будет стоить?" | 30 мин |
| Запрос КП | "Пришлите КП" | 30 мин |
| Предложение встречи | "Давайте встретимся" | 30 мин |
| Жалоба / угроза | "Вы обещали вчера!" | **10 мин** |
| Вопрос без фактов в БД | recall вернул пусто | 30 мин |
| Любой запрос требующий суждения | "Что предлагаете?" | 30 мин |

### Алгоритм

1. Ничего не советовать клиенту
2. Записать в memory (label=risk или task)
3. Написать клиенту: "Поняла, передала Камилю. Скоро ответит!" (или вариации)
4. Создать pending notification в БД
5. Запустить таймер (async sleep)
6. Проверить: owner ответил за это время?
   - Да → cancel notification
   - Нет → уведомить owner'у в личку бота

### Night Mode (23:00-08:00 GMT+8)

| Ситуация | Действие |
|----------|---------|
| Escalation (не complaint) | Отложить до 08:00 GMT+8 |
| Complaint (жалоба, risk) | Отправить СРАЗУ (10 мин), даже ночью |
| Fact response | Отправлять как обычно (клиент ждёт) |
| Client reminders | Не отправлять (только 09:00-20:00) |
| Proactive owner alerts | Отложить до 08:00 |

## 🛡 ANTI-SPAM (ResponseGuard)

Защита от спам-ответов и петель.

```
Правило 1: min_interval = 180с
  Не отвечать чаще раза в 3 минуты в один чат.

Правило 2: max_consecutive = 3
  После 3 ответов подряд → cooldown 15 минут.

Правило 3: cooldown = 900с
  Circuit breaker. Сбрасывается при паузе >10 минут.

Правило 4: default = skip
  При сомнениях — не отвечать.
```

**Реализация:** Хранить в memory/state:
```json
{
  "chat_id": {
    "last_response": 1711891200,
    "consecutive": 2,
    "cooldown_until": null
  }
}
```

## 📋 PROGRESS CHECK-IN

### Flow

```
Day 0: Появляется задача "Сделать КП" с due_at = Day+5
Day 2 10:00 GMT+8: Proactive check-in
  → "📋 Check-in: Сделать КП для Ивана. Дедлайн через 3д. Как дела? [task:42]"
  → Owner: "в работе"
  → LLM парсит: {"action": "in_progress"}
  → UPDATE tasks SET status='in_progress'

Day 4 10:00: Check-in
  → "📋 Check-in: Сделать КП. Дедлайн завтра. Статус: in_progress."
  → Owner: "готово, клиент подписал"
  → LLM парсит: {"action": "done", "notes": "клиент подписал"}
  → UPDATE tasks SET status='done'
```

### Парсинг ответа owner'а

LLM определяет action:

| Action | Что делать | SQL |
|--------|-----------|-----|
| `done` | Задача выполнена | status='done' |
| `in_progress` | В работе | status='in_progress' |
| `extend` | Продлить срок | due_at = new_date или +N дней |
| `blocked` | Есть блокер | priority='urgent' |
| `cancel` | Отмена | status='cancelled' |
| `remind_tomorrow` | Напомнить завтра | (через scheduler) |
| `remind_date` | Напомнить в дату | (через scheduler) |
| `escalate` | Проблема, нужна помощь | (уведомить) |
| `update` | Просто обновление | notes += "новый статус" |

### Check-in сообщение (шаблон)

```
📋 Check-in: {description}

👤 Клиент: {contact_name}
💰 Сделка: {amount} ({stage})
📅 Дедлайн: {due_date} (осталось {days_left}д)
📌 Статус: {status}

Последнее в чате:
{last_messages}

Как дела? Напишите «готово», «в работе», или опишите статус.
[task:{task_id}]
```

## 🔍 FOLLOW-UP DETECTION

### Когда срабатывает

LLM ищет implicit obligations в тексте:

```
"Договорились, пришлю КП в среду"
  → obligation: "пришлёт КП", due: среда, who: owner

"Клиент сказал оплатит до пятницы"
  → obligation: "клиент оплатит", due: пятница, who: client

"Напишу ему завтра"
  → obligation: "написать клиенту", due: завтра, who: owner

"Ладно, тогда в понедельник обсудим"
  → obligation: "обсудить", due: понедельник, who: both
```

### Результат → task + reminder

1. Создаёт task с due_at
2. Standard proactive engine подхватит через upcoming tasks
3. who=owner → proactive owner alert за 2 дня
4. who=client → client reminder за 2 дня

## 📨 CLIENT REMINDERS

### Когда напоминать клиенту

Только obligations клиента (то, что КЛИЕНТ должен сделать):

| Obligation | За сколько | Шаблон |
|-----------|-----------|--------|
| Неоплаченный счёт | 2 дня | "Напоминаем об оплате..." |
| Неподписанный договор | 3 дня | "Напоминаем о подписании..." |
| Обещанные материалы | 2 дня | "Напоминаем о материалах..." |

### Как отправлять

1. **Через business API** (can_reply=True) — от имени owner'а
2. **Fallback:** owner'у → owner пишет сам
3. **Night mode:** только 09:00-20:00 GMT+8
4. **Dedup:** не отправлять повторно за 3 дня

### Шаблоны

```
Счёт: "Добрый день! 🧾 Напоминаем об оплате по договору «{summary}». Сумма: {amount}. Срок: {due_date}. Если уже оплатили — проигнорируйте."

Договор: "Добрый день! 📝 Напоминаем о подписании договора «{summary}». Срок: {due_date}. Если есть вопросы — напишите."

Задача: "Добрый день! 📋 Напоминаем: {description}. Срок: {due_date}. Если уже сделали — проигнорируйте."
```
